# java-janggi

# What is this?

장기 step2 미션 구현

## 기능

- [x] 데이터베이스에 연동되어 턴마다 게임 세이브 기능 지원됩니다
    - 중간에 데이터베이스가 끊겨도 게임은 진행 가능합니다.
- [x] 장기를 구현하였습니다
    - 플레이어는 2명으로 홍,청으로 플레이어하게 됨
    - 플레이어는 좌표를 선택하여 기물을 움직일 수 있음
    - 유효하지 않은 움직임을 입력할 시 에러 메시지가 출력되며 다시 입력받음
    - 둘중의 한명의 왕이 죽으면 게임이 종료됨
    - 게임이 종료되면 각 팀들의 점수가 출력됩니다.
    - 게임이 종료되면 저장되었던 게임은 삭제됩니다
- [x] 이제 궁성 내 기물들의 움직임도 제대로 동작합니다!
- [x] 이제 궁은 가만히 있을 수 있습니다!

## 실행 및 테스트 방법

- 데이터베이스는 docker-compose를 이용하여 배포할 수 있습니다.
  https://www.docker.com/get-started/
- 그 뒤 애플리케이션을 실행하여 게임을 시작할 수 있습니다

```
./start-game.sh
```

## 문제 해결방법

- 만약 스크립트 쉘 실행이 안된다면 권한을 상승시켜서 실행시켜주세요!
- 만약 지속적으로 "장기 저장 기능이 동작하지않습니다"가 출력된다면 ./initiateDatabase.sh를 실행시켜주세요.

## 스스로 생각했을때 원하는, 목표하는 객체지향의 형태

- 제 이상적인 객체지향은 빨랐으면 좋겠습니다
- 새로운 기능이 생길때, 버그를 찾아야할때 등 코드의 명확성이 받혀주면 생각해야할 컨텍스트가 작아져서 그만큼 빨리 작업할 수 있다고 생각합니다. 제가 원하는 객체지향의 형태도 그러합니다.
- 이번에 새 요구사항을 구현하면서 머리속에서 예상하는 바뀌어야할 요소들과 객체들이 거의 1대1로 맵핑되는 것을 상상했고 기능 구현단에서는 크게 벗어나지 않아서 성공적이었던 것 같습니다.
- 아직 대대적인 리팩토링을 시도하지 않았습니다.
    - 이에 코드스멜이 느껴지는 부분을 오리와 함께 이야기해보고 그 과정에서 체화가 안된것들, 통일되지 않는 가치들을 합쳐보고 싶습니다!

# 궁금한점!

명확성을 위해서 다음의 생각을 가지고 구현해보았는데 어떻게 생각하시는지 궁금합니다!

- 궁성 위치를 Position이 가지고 있고, 궁성안에 있는지에 따라서 각 기물들의 움직임을 정의한다
    - 이때 발생하는 중복코드는 같은 책임에서 바뀌기 때문에 유지보수에 문제가 없다
    - 세부적인 상황) 차와졸은 수직으로 움직이는 객체지만, 수직이라는 움직임을 각각 인터페이스 등으로 묶을 필요 없음
      그 이유는 수직,수평 이동을 추상화했을때 이점이 없기 때문. 오히려 차를 수정해야할때 다른 인터페이스까지 봐야하기 때문에 복잡도만 커짐
      또한, 인터페이스로 묶었던 것을 다시 풀어야할 수도 있음. 즉 중복코드 - 유지보수를 위해서 지양하는것
    - 명확한 코드를 위해서 추상화는 필요할때만, 필요로하는 클래스는 최소화
- > 필요로하는 클래스는 최소화

  단, 상수들의 경우 Enum으로 관리가 편하게한다

## 데이터베이스 테스트

- dao와 사용 서비스를 이용하여 테스트 하였습니다.
- 테스트는 모킹, 스텁 방식을 고려하였는데, 두 방식 모두 외부 의존성을 위한 많은 테스트, 클래스가 필요했었습니다.
- 이에 테스트 db를 도입하는 것을 선택하였는데, 이 경우 데이터베이스 서버에 부하가 갈 수 있을 것 같습니다.
  현재는 부하가 크지 않을 것 같아 이 방식을 선택해보았는데,
  스텁 방식으로도 구현해보고 싶습니다.
- 데이터베이스에 값을 추가, 그것을 가져올 수 있는지 테스트를 작성하는 과정에서 JUnit Order 기능을 사용했습니다.
  이러면 각 테스트들끼리 순서 의존성이 생기는 문제가 생길 수 있을 것 같아 중복 코드가 발생하여도 Order 기능을 사용하지 않고 진행하였습니다.

## 아쉬운 점

- 믿고 사용할 수 있는 방어적 설계가 잘 안되는 것 같습니다
    - 어디까지 고려를 해야하는지, 전방위적으로 방어가 잘 되면 좋겠지만, 현실적으로 타협을 하는 편인것 같습니다.
    - 예로, PlayerPieces 생성자에서 받는 Pieces는 유효한 장기말들이 맞는지 모두 확인을 해야 안전하게 사용가능할것 같습니다.
    - 여기서, 장기에서 사용할 Pieces들을 만드는 책임은 현재 InitiateJanggiTeamPieces에서 하고 있으니 검증하지 않아도 되는 것일까요?
    - 만약 그렇지 않다면 모든 Pieces를 사용하는 부분에서 검증을 해야하는것이 되어서 어떻게 해결해야할지 스스로 생각을 더 해봐야할 것 같습니다.
